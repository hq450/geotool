# geotool 设计文档

## 1. 设计目标

`geotool` 是一个面向 geosite 文件的轻量级解析工具，重点解决两个问题：

1. 快速列出 geosite 文件中的全部分类
2. 按分类导出原始规则，并保留规则类型与属性信息

设计时的核心约束如下：

- 二进制体积尽量小，目标为 `100KB` 级别
- 依赖尽量少，避免引入大型第三方库
- 方便后续交叉编译到 `armv7`、`aarch64` 等平台
- 输出结果必须尽量接近 geosite 中的原始明文规则语义

## 2. 为什么选择 Zig

相比 C，Zig 更适合这个项目的当前阶段。

主要原因：

- Zig 对静态编译和交叉编译支持更直接
- 标准库已经足够完成文件读取、命令行解析、字符串处理和容器管理
- 手写 protobuf 读取器时可以保持较低复杂度
- 在保证开发效率的同时，仍能得到很小的可执行文件

虽然 C 在极端手工优化下仍可能进一步压缩体积，但工程维护成本会明显上升，尤其是在多平台构建场景下。

## 3. 输入数据模型

本项目处理的输入包括：

- `v2fly/domain-list-community` 生成的 `dlc.dat`
- `Loyalsoldier/v2ray-rules-dat` 生成的 `geosite.dat`

它们都属于 geosite 类数据文件，本质上是 protobuf 编码的规则集合。

从当前实现视角，可抽象为：

```text
GeoSiteList
  └── GeoSite
        ├── category / code
        └── Domain
              ├── type
              ├── value
              └── Attribute[]
```

其中：

- `GeoSite` 对应一个分类
- `Domain` 对应一条规则
- `type` 决定输出前缀
- `Attribute` 决定附加的 `@attr` 或 `@key=value`

## 4. 功能需求

### 4.1 分类列表

输入一个 geosite 文件，程序应输出全部分类名，每行一个。

要求：

- 能同时兼容 `country_code` 和 `code` 两种分类字段来源
- 输出顺序稳定，当前实现按不区分大小写排序输出

### 4.2 分类导出

输入 geosite 文件和一个或多个分类名，程序应输出该分类下的全部规则。

要求：

- 保留规则类型
- 保留属性
- 分类名匹配不区分大小写
- 支持逗号分隔的多分类
- 多分类合并时去除重复规则

### 4.3 规则导出格式

输出格式定义如下：

```text
<type>:<value> [@attr] [@key=value]
```

示例：

```text
domain:google.com
full:firebase.google.com
keyword:google
regexp:^example$
domain:test.com @ads
full:api.example.com @region=cn
```

## 5. 总体架构

当前项目采用三层结构：

### 5.1 CLI 层

文件：[src/main.zig](/home/sadog/project/geotool/src/main.zig)

职责：

- 解析命令行参数
- 读取输入文件
- 解析 `-c` 中的多分类列表
- 调用 geosite 解析层
- 处理帮助信息和错误输出

### 5.2 geosite 解析层

文件：[src/geosite.zig](/home/sadog/project/geotool/src/geosite.zig)

职责：

- 遍历 protobuf 数据结构
- 解析分类名
- 导出规则
- 多分类合并
- 规则去重
- 属性格式化

这是项目的核心逻辑层。

### 5.3 protobuf 最小读取层

文件：[src/pb.zig](/home/sadog/project/geotool/src/pb.zig)

职责：

- 读取 protobuf field tag
- 读取 varint
- 读取 length-delimited 字段
- 跳过不关心的字段

当前只实现本项目所需的最小子集，不追求通用 protobuf 解码器能力。

## 6. 关键设计决策

### 6.1 不引入完整 protobuf 依赖

原因：

- 目标是极小体积
- geosite 使用到的 protobuf 特性非常有限
- 只需要支持 `varint`、`length_delimited`、`fixed32`、`fixed64` 的跳过逻辑

因此项目采用手写最小读取器，而不是链接完整 protobuf 运行时。

### 6.2 导出时保留规则类型

用户需求不是“导出域名列表”，而是“导出原始规则语义”。

因此不能只输出：

```text
google.com
```

必须输出：

```text
domain:google.com
full:mail.google.com
keyword:google
regexp:^gstatic\.
```

这也是该工具相对普通域名导出脚本的关键区别。

### 6.3 多分类去重按完整规则文本进行

去重不是只比较 `value`，而是比较最终导出的整行文本。

也就是说：

- `domain:example.com`
- `full:example.com`

这两条不应被当成重复。

同样：

- `domain:example.com @ads`
- `domain:example.com`

也不应被当成重复。

因此当前实现以最终输出的完整规则字符串作为去重键。

### 6.4 分类存在性先校验

在真正输出前，先扫描一遍输入文件，确认用户指定的所有分类都存在。

这样做的好处：

- 可以避免输出了一半后才发现有分类不存在
- 对多分类导出更符合“要么全部成功，要么失败”的行为预期

当前策略是：

- 只要指定列表中有任意分类不存在，整体返回 `category not found`

## 7. 规则类型映射

当前实现中的规则类型映射如下：

```text
0 -> keyword
1 -> regexp
2 -> domain
3 -> full
```

若后续上游格式扩展出新的类型，当前实现会输出：

```text
unknown:<value>
```

这保证了解析器在未知类型出现时不会直接崩溃。

## 8. 多分类导出流程

以命令：

```bash
./zig-out/bin/geotool export -i geosite.dat -c GFW,AI,GOOGLE
```

为例，内部流程如下：

1. CLI 层读取 `-c` 参数
2. 将字符串按逗号拆分
3. 去掉每个分类名前后的空白
4. 按大小写不敏感规则去除重复分类名
5. 解析 geosite 文件并校验所有分类都存在
6. 按用户给出的顺序依次导出每个分类
7. 将每条规则格式化成完整文本
8. 以完整文本作为 key 做去重
9. 输出最终结果

## 9. 时间复杂度与空间复杂度

### 9.1 list

主要过程是一次顺序扫描 geosite 文件。

时间复杂度：

- 解析阶段约为 `O(N)`
- 分类排序额外为 `O(C log C)`

其中：

- `N` 为文件大小或字段数量规模
- `C` 为分类数量

### 9.2 export

当前实现中，多分类导出采用“存在性预检查 + 按分类扫描”的策略。

因此在最坏情况下，复杂度近似为：

```text
O(N * K)
```

其中：

- `N` 为 geosite 数据规模
- `K` 为去重后的分类数量

这个策略的优点是实现简单、二进制小、行为明确。

如果未来发现多分类数量非常大，可以考虑改成单次扫描并建立分类索引，但那会引入更多内存和代码复杂度。

### 9.3 去重开销

规则去重使用字符串集合保存已输出的完整规则。

空间复杂度约为：

```text
O(R)
```

其中 `R` 为最终输出后的唯一规则数。

## 10. 错误处理策略

当前错误处理原则是“简单、直接、足够定位问题”。

主要错误包括：

- 参数错误
- 分类不存在
- protobuf 结构异常
- 输入文件截断或损坏
- 内存分配失败

CLI 层对常见错误给出简短输出：

- 参数错误时显示帮助
- 分类不存在时输出 `category not found`

更底层的 protobuf 或 IO 错误则直接输出错误名。

## 11. 测试策略

当前测试主要集中在 [src/geosite.zig](/home/sadog/project/geotool/src/geosite.zig)。

测试方式不是依赖外部真实 geosite 文件，而是在测试中手工构造最小 protobuf 数据。

这样做的优点：

- 测试稳定，不依赖外部样本文件
- 可精确控制边界条件
- 更适合交叉平台回归

当前已覆盖：

- 分类列出
- 单分类导出
- 分类不存在
- 多分类合并
- 重复分类去重
- 重复规则去重

## 12. 当前限制

当前实现仍有一些明确限制：

- 错误信息偏简洁，没有指出具体哪个分类不存在
- 多分类导出是按分类多次扫描输入文件，不是单次建索引
- 未提供正则校验，仅做原样输出
- 未提供批量导出所有分类为多个文件的模式
- 未提供 JSON、CSV 等其他输出格式

这些限制是有意保留的，目的是优先保持实现简单、体积小、可维护。

## 13. 后续演进建议

如果项目后续继续扩展，可以考虑以下方向：

### 13.1 用户体验

- 在分类不存在时输出具体缺失的分类名
- 增加 `search` 子命令，支持按关键词查分类
- 增加 `count` 子命令，输出分类规则数量

### 13.2 性能

- 增加单次扫描模式，先建立分类偏移索引，再执行导出
- 对超大 geosite 文件考虑 mmap 或流式接口

### 13.3 输出能力

- 支持 JSON 输出
- 支持批量导出全部分类
- 支持属性过滤

### 13.4 构建与发布

- 增加 `armv7-linux-musleabihf` 构建目标
- 增加 `aarch64-linux-musl` 构建目标
- 补充发布脚本与 CI 产物打包

## 14. 总结

`geotool` 的设计重点不是“做一个通用规则平台”，而是：

- 专注 geosite 解析
- 输出规则语义完整
- 保持实现小而直接
- 为后续多平台静态发布做准备

当前这套设计已经满足最初需求，并且保留了继续向多平台、小体积工具演进的空间。
